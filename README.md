## 📃 Цель проекта
Проект направлен на разработку рекомендательной системы для банка «Стар».  
Ключевая цель — предоставить клиентам **персонализированные предложения** по банковским продуктам и услугам на основе:  
- статических правил,  
- динамических правил,  
- анализа транзакционной активности клиентов.  

Система также предоставляет инструменты для управления правилами и сбора статистики.

---

## ⚙ Технологический стек
- **Java 21, Spring Boot** — основной фреймворк.  
- **PostgreSQL** — база данных для хранения правил.  
- **H2 (readonly)** — демо-база для клиентских данных.  
- **Spring Data JDBC / JPA** — работа с БД.  
- **Liquibase** — миграции БД.  
- **Caffeine Cache** — кэширование SQL-результатов.  
- **Telegram Bot API** — интеграция с мессенджером.  
- **JUnit 5, Mockito** — тестирование.  
- **Swagger / OpenAPI** — документация REST API.  

---

## 💡 Основные возможности
- Выдача рекомендаций клиентам по запросу (`/recommendation/{userId}`).  
- Поддержка **динамических правил**, добавляемых через API (`/rule`).  
- Поддержка **статических правил** (первое ТЗ).  
- Управление кэшем и сервисной информацией (`/management/*`).  
- Получение рекомендаций через **Telegram-бота**.  
- Сбор статистики по срабатыванию правил.

## 📘 User Stories
-### US-1. Получение рекомендаций
**Как пользователь**  
Я хочу получать рекомендации по банковским продуктам на основе моих транзакций,  
чтобы я мог выбирать наиболее подходящие предложения.  

### US-2. Управление правилами
**Как менеджер**  
Я хочу добавлять, изменять и удалять динамические правила рекомендаций через REST API,  
чтобы я мог быстро адаптировать систему к изменяющимся бизнес-требованиям.  

### US-3. Статистика по правилам
**Как менеджер**  
Я хочу просматривать статистику срабатывания правил,  
чтобы понимать, какие правила наиболее эффективны.  

### US-4. Telegram-бот
**Как пользователь**  
Я хочу получать рекомендации через Telegram-бота,  
чтобы иметь доступ к сервису прямо из мессенджера.  

### US-5. Управление сервисом
**Как внешняя система (или DevOps)**  
Я хочу иметь возможность сбрасывать кэш и получать информацию о сервисе через `/management/*`,  
чтобы поддерживать стабильность работы приложения.

## 🔧 Нефункциональные требования

- **NFR-1**: Время отклика < 500 мс при 1000 RPS.  
- **NFR-2**: Масштабируемость: система должна поддерживать горизонтальное масштабирование.  
- **NFR-3**: Логирование всех входящих запросов и ошибок.  
- **NFR-4**: Покрытие unit-тестами не менее 80% бизнес-логики.  
- **NFR-5**: Конфигурация приложения должна задаваться через переменные окружения.
